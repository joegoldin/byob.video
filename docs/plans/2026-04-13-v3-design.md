# byob.video v3.0.0 Design Spec

## Goals

Two tracks for the v3.0.0 release:

1. **Architecture cleanup** â€” break apart the monolithic files, improve persistence, lay groundwork for future scaling
2. **YouTube comments panel** â€” show YouTube comments below the video player, always visible, fetched via Data API v3

Refactor ships first, then the comments feature is built on the clean architecture.

## Track 1: Architecture Cleanup

### 1.1 Split room_live.ex (~1200 lines)

Current `room_live.ex` is a monolith handling mount, URL preview, playback events, queue management, PubSub handlers, and the full template. Split into focused modules:

| Module | Responsibility |
|--------|----------------|
| `RoomLive` | Mount, assigns, top-level `render/1` delegating to components |
| `RoomLive.UrlPreview` | `handle_event` for URL paste, oembed preview, play-now/queue-from-preview |
| `RoomLive.Playback` | `handle_event` for play, pause, seek, video_ended, sync corrections |
| `RoomLive.Queue` | `handle_event` for queue add, remove, reorder, skip, history replay |
| `RoomLive.Sidebar` | Function components: queue list, history list, activity log, users card |
| `RoomLive.PubSub` | All `handle_info` callbacks for PubSub broadcasts (queue_updated, sync_play, sync_pause, etc.) |

**Approach**: Use Phoenix LiveView's built-in support for splitting event handlers. Each module defines `handle_event/3` or `handle_info/2` clauses, and `RoomLive` delegates via pattern matching or `use` macros. The template uses function components from `RoomLive.Sidebar` for the sidebar sections.

**Key constraint**: All modules share the same socket/assigns. No separate LiveView processes â€” this is a code organization refactor, not an architectural change.

### 1.2 Split video_player.js (~1050 lines)

Current `video_player.js` mixes the Phoenix hook lifecycle, sync engine, three player types, and SponsorBlock. Split into ES modules:

| Module | Responsibility |
|--------|----------------|
| `hooks/video_player.js` | Phoenix hook lifecycle (`mounted`, `destroyed`), event handler registration, delegates to modules |
| `sync/engine.js` | Clock sync (NTP probes), reconciliation loop, drift correction, generation counters, event suppression |
| `players/youtube.js` | YouTube IFrame API: load, play, pause, seek, state change handling, error detection |
| `players/direct.js` | HTML5 `<video>` element: load, play, pause, seek, ended detection |
| `players/extension.js` | Extension placeholder UI, polling, media info display |
| `sponsor_block.js` | Segment data management, seek bar injection, auto-skip logic |

**Player interface**: Each player module exports a consistent interface:

```javascript
{ load(id, url), play(), pause(), seek(t), destroy(), getCurrentTime(), getDuration(), getState() }
```

`video_player.js` calls the interface without knowing which player is active.

### 1.3 SQLite persistence improvements

**Current problem**: Room state is serialized as an opaque Erlang binary via `term_to_binary/binary_to_term`. No schema versioning â€” adding/removing fields requires careful backward compatibility or manual DB wipes.

**Solution**:

- Add `schema_version INTEGER DEFAULT 1` column to the `rooms` table
- On load, check version and run migration functions sequentially: `migrate_1_to_2(state)`, `migrate_2_to_3(state)`, etc.
- Each migration transforms the state map (add fields with defaults, rename keys, remove deprecated fields)
- Current state format becomes version 1; the v3.0.0 refactor bumps to version 2
- Migration module: `Byob.DB.Migrations` with numbered functions

**Not changing**: Still using `term_to_binary` for serialization (it works fine). The versioning just ensures we can evolve the state shape safely.

### 1.4 Multi-instance groundwork

**Not shipping multi-node in v3.0.0**, but removing hard single-instance assumptions:

- Audit PubSub usage â€” ensure all room communication goes through `Phoenix.PubSub` (already mostly true)
- Document what would need to change for distributed Erlang or Redis PubSub adapter
- Ensure room GenServer lookup doesn't assume local-only Registry
- No immediate changes to Fly.io config (stays at `max_machines_running = 1`)

## Track 2: YouTube Comments Panel

### 2.1 Overview

Always-visible scrollable comments section below the YouTube player. Shows top-level comments sorted by relevance. Fetched server-side via YouTube Data API v3. Hidden entirely for non-YouTube sources or when API quota is exhausted.

### 2.2 Server: Byob.YouTube.Comments

New module for fetching and caching YouTube comments.

**Fetching**:
- Calls `GET https://www.googleapis.com/youtube/v3/commentThreads`
  - Parameters: `part=snippet`, `videoId=VIDEO_ID`, `order=relevance`, `maxResults=20`, `textFormat=plainText`
  - Auth: `key=YOUTUBE_API_KEY` (env var, already set as Fly secret)
- Returns list of comment maps: `%{author, author_avatar, text, likes, reply_count, published_at}`
- Pagination: stores `nextPageToken` from response, used for "load more" requests

**Caching**:
- ETS table `:youtube_comments_cache`
- Key: video ID
- Value: `{comments, next_page_token, fetched_at}`
- TTL: 15 minutes â€” comments don't change fast
- Cache is per-instance (not shared across rooms). If two rooms watch the same video, second one hits cache.

**Quota handling**:
- YouTube Data API quota: 10,000 units/day. `commentThreads.list` costs 1 unit per request.
- If API returns HTTP 403 with `quotaExceeded` reason, set `Application.put_env(:byob, :youtube_quota_exhausted, true)`
- All subsequent fetch calls check this flag and return `{:error, :quota_exhausted}` immediately
- Flag resets daily (checked against UTC date) or on app restart
- No error UI â€” the comments section simply doesn't render

**Disabled comments**:
- Some videos have comments disabled. API returns empty list or error. Treat same as no comments â€” don't render the panel.

### 2.3 Server: RoomServer integration

When a video starts playing (or changes):
- If source type is `:youtube`, the RoomServer spawns a Task to fetch comments
- On success, broadcasts `{:comments_updated, %{video_id: id, comments: list, next_page_token: token}}`
- RoomLive handles this in PubSub module, assigns `@comments` and `@comments_next_page`

Pagination ("load more"):
- LiveView sends `"comments:load_more"` event
- Handler calls `Byob.YouTube.Comments.fetch(video_id, page_token: next_token)`
- Appends new comments to `@comments`, updates `@comments_next_page`

### 2.4 Frontend: Comments component

New LiveView function component, rendered below the player div in the main template.

**Renders when**: `@comments` is non-nil and non-empty, and current source type is YouTube.

**Layout** (matches approved mockup â€” Option B: Always Visible):
- Fixed-height container (~200px), scrollable, below the video player
- Header: "Comments" label + count badge
- Comment rows: avatar (28px circle), author name, relative timestamp, comment text, like count, reply count
- Bottom: "Load more" button (if `@comments_next_page` is not nil)
- Fade/gradient at bottom edge to hint scrollability

**Styling**: Matches existing dark theme. Uses daisyUI/Tailwind classes consistent with sidebar components.

**Non-YouTube videos**: Component not rendered. No empty state, no "comments unavailable" â€” just absent.

### 2.5 Environment

- `YOUTUBE_API_KEY` â€” required for comments feature. If not set, comments panel never renders (feature silently disabled).
- Added to CLAUDE.md environment variables section.
- Already set as Fly secret.

## What's NOT in scope

- Replying to YouTube comments from byob
- Nested reply threads (only top-level comments)
- Comment search or filtering
- Syncing comment scroll position across room users
- Chat/messaging between byob users (separate future feature)
- Horizontal scaling / multi-node deployment
- New video source types
- User accounts or persistent identity
- Mobile app
